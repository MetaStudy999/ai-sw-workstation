# 포트 입문 가이드

이 문서는 포트 관리를 처음 배우는 사람이
`포트 관련 명령어`, `실행 예제`, `실제로 응용하는 순서`를
쉽게 이해할 수 있도록 정리한 입문용 안내서입니다.

실습은 macOS/Linux의 `zsh` 또는 `bash` 기준으로 설명합니다.
예제는 이 저장소 루트 디렉터리에서 실행한다고 가정합니다.

핵심 목표는 다음과 같습니다.

- 포트가 무엇인지 감을 잡는다.
- 어떤 프로그램이 어떤 포트를 쓰는지 확인할 수 있다.
- 포트가 열려 있는지, 접속이 되는지 테스트할 수 있다.
- 포트 충돌이 났을 때 어떤 순서로 확인해야 하는지 이해한다.
- Docker와 포트 매핑의 관계를 함께 이해할 수 있다.

---

## 1. 포트를 한 줄로 이해하기

포트는 **프로그램이 네트워크 요청을 받는 번호가 붙은 문**이라고 생각하면 쉽습니다.

예를 들어:

- `http://localhost:8080`
- `http://localhost:3000`
- `https://localhost:443`

위 주소에서 중요한 것은 `8080`, `3000`, `443` 같은 숫자입니다.
이 숫자가 바로 포트 번호입니다.

쉽게 말하면:

- 프로그램은 특정 포트에서 요청을 기다리고
- 브라우저나 `curl`은 그 포트로 접속을 시도합니다

그래서 웹 서버가 `8080` 포트에서 실행 중이면,
우리는 `http://localhost:8080`으로 접속해야 합니다.

---

## 2. 먼저 알아두면 쉬운 핵심 개념 6가지

### 2-1. `localhost`

`localhost`는 "지금 내가 쓰고 있는 내 컴퓨터 자신"을 뜻합니다.

즉:

- `http://localhost:8080`

은 "내 컴퓨터의 8080 포트에 접속한다"는 뜻입니다.

### 2-2. 프로그램은 포트를 점유한다

어떤 프로그램이 이미 `8080` 포트를 사용 중이면,
다른 프로그램이 같은 포트를 바로 쓰지 못하는 경우가 많습니다.

이때 흔히 보는 오류가 "포트가 이미 사용 중"이라는 메시지입니다.

### 2-3. 포트가 열려 있어야 접속된다

브라우저 주소를 맞게 입력해도,
실제로 그 포트에서 프로그램이 요청을 받고 있지 않으면 접속이 실패합니다.

즉, 아래 2가지가 함께 맞아야 합니다.

- 프로그램이 실행 중이어야 함
- 그 프로그램이 원하는 포트에서 기다리고 있어야 함

### 2-4. 호스트 포트와 컨테이너 포트는 다를 수 있다

Docker에서는 특히 이 개념이 중요합니다.

예:

```bash
docker run -p 8080:80 nginx:alpine
```

의미:

- 내 컴퓨터(호스트)의 `8080` 포트로 들어온 요청을
- 컨테이너 내부의 `80` 포트로 전달

즉, 브라우저는 `localhost:8080`으로 접속하지만
실제 웹 서버는 컨테이너 안의 `80` 포트에서 동작할 수 있습니다.

### 2-5. 포트 확인과 포트 테스트는 다르다

초보자가 자주 헷갈리는 부분입니다.

- `lsof`: 누가 그 포트를 쓰는지 확인
- `curl`: 실제 HTTP 응답이 오는지 확인
- `nc`: 해당 포트에 연결 자체가 되는지 확인

즉:

- `lsof`는 "누가 쓰고 있나?"
- `curl`은 "웹 서버가 잘 응답하나?"
- `nc`는 "문이 열려 있나?"

처럼 이해하면 편합니다.

### 2-6. 이 프로젝트에서 포트가 설정된 위치

이 저장소에서는 포트 관련 설정을 아래 파일에서 볼 수 있습니다.

- [`.env`](./.env): `HOST_PORT=8080`
- [`docker-compose.yml`](./docker-compose.yml): `"${HOST_PORT}:80"`
- [`compose/nginx/default.conf`](./compose/nginx/default.conf): `listen 80;`

즉, 이 프로젝트는 기본적으로:

- 호스트 포트: `8080`
- 컨테이너 포트: `80`
- NGINX가 실제로 듣는 포트: `80`

흐름으로 이해하면 됩니다.

## 3. 입문자가 먼저 익힐 포트 관련 명령어

| 목적 | 명령어 | 의미 |
| --- | --- | --- |
| 현재 설정된 포트 확인 | `cat .env` | 프로젝트에서 사용할 호스트 포트 확인 |
| 웹 접속 테스트 | `curl http://localhost:8080` | 실제 HTTP 응답 확인 |
| 헬스체크 테스트 | `curl http://localhost:8080/health` | 서버가 살아 있는지 간단히 확인 |
| 포트 사용 여부 확인 | `lsof -i :8080` | 어떤 프로세스가 8080을 쓰는지 확인 |
| 리슨 중인 TCP 포트 확인 | `lsof -nP -iTCP:8080 -sTCP:LISTEN` | 실제로 8080에서 대기 중인 프로세스 확인 |
| 연결 가능 여부 확인 | `nc -vz localhost 8080` | 해당 포트에 연결이 되는지 확인 |
| PID 상세 확인 | `ps -p PID -o pid,ppid,command=` | 해당 프로세스가 무엇인지 확인 |
| 프로세스 종료 | `kill PID` | 정상 종료 요청 |
| 강제 종료 | `kill -9 PID` | 마지막 수단으로 강제 종료 |
| 컨테이너 확인 | `docker ps` | 실행 중인 컨테이너와 포트 매핑 확인 |
| Compose 상태 확인 | `docker compose ps` | Compose 서비스 상태와 포트 확인 |
| 컨테이너 포트 확인 | `docker port 컨테이너명` | 컨테이너의 포트 매핑 확인 |

## 4. 명령어별 가장 쉬운 예제

### 4-1. `cat .env`

먼저 "이 프로젝트가 원래 어떤 포트를 쓰기로 했는지" 확인합니다.

```bash
cat .env
```

예를 들어 아래처럼 보이면:

```bash
HOST_PORT=8080
APP_ENV=compose-dev
```

지금 이 프로젝트는 기본적으로 `8080` 포트를 사용할 예정이라는 뜻입니다.

### 4-2. `curl`

브라우저를 열기 전에 터미널에서 가장 빠르게 접속을 확인하는 방법입니다.

```bash
curl http://localhost:8080
```

헬스체크 경로까지 확인할 수도 있습니다.

```bash
curl http://localhost:8080/health
```

예상되는 결과 예시:

```bash
ok
```

초보자 팁:

- 브라우저보다 먼저 `curl`로 확인하면 문제를 빨리 좁힐 수 있습니다
- `curl`이 안 되면 브라우저도 거의 안 됩니다

### 4-3. `lsof`

어떤 프로그램이 특정 포트를 사용 중인지 확인합니다.

```bash
lsof -i :8080
```

조금 더 정확하게 "지금 리슨 중인 TCP 포트"만 보고 싶다면:

```bash
lsof -nP -iTCP:8080 -sTCP:LISTEN
```

예를 들어 아래처럼 보일 수 있습니다.

```bash
COMMAND   PID   USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
Docker   1234   me    120u  IPv4 0x...      0t0  TCP *:8080 (LISTEN)
```

여기서 초보자는 아래 3가지만 보면 충분합니다.

- `COMMAND`: 어떤 프로그램인지
- `PID`: 프로세스 번호
- `NAME`: 어떤 포트인지

### 4-4. `nc`

HTTP 응답까지는 아니어도,
"해당 포트에 연결 자체가 되는지" 빠르게 테스트할 수 있습니다.

```bash
nc -vz localhost 8080
```

성공 예시:

```bash
Connection to localhost port 8080 [tcp/http-alt] succeeded!
```

이 명령은 웹 서버뿐 아니라 여러 TCP 포트 점검에 응용하기 좋습니다.

### 4-5. `ps`

`lsof`에서 확인한 PID가 정확히 어떤 프로그램인지 더 자세히 보고 싶을 때 사용합니다.

```bash
ps -p 1234 -o pid,ppid,command=
```

의미:

- `pid`: 현재 프로세스 번호
- `ppid`: 부모 프로세스 번호
- `command`: 실제 실행 명령

즉, `1234`가 정말 Docker인지, Node 서버인지, 다른 앱인지 더 분명하게 볼 수 있습니다.

### 4-6. `kill`

충돌 중인 프로세스를 종료할 때 사용합니다.

먼저 일반 종료를 시도합니다.

```bash
kill 1234
```

그래도 안 꺼질 때만 마지막 수단으로 강제 종료합니다.

```bash
kill -9 1234
```

주의:

- `kill -9`는 가장 강한 방법이므로 먼저 쓰지 않는 것이 좋습니다
- 중요한 프로그램을 잘못 종료하지 않도록 `ps`로 먼저 확인하는 습관이 좋습니다

### 4-7. `docker ps`

Docker에서는 어떤 컨테이너가 어떤 포트로 열려 있는지 먼저 확인합니다.

```bash
docker ps
```

예를 들어 포트 부분에 아래처럼 보일 수 있습니다.

```text
0.0.0.0:8080->80/tcp
```

의미:

- 호스트 `8080`
- 컨테이너 `80`

로 연결되었다는 뜻입니다.

### 4-8. `docker port`

특정 컨테이너의 포트 매핑만 간단히 보고 싶을 때 편합니다.

```bash
docker port compose-web
```

예시:

```bash
80/tcp -> 0.0.0.0:8080
```

### 4-9. `docker compose ps`

이 프로젝트처럼 Compose를 쓸 때는 서비스 상태를 이 명령으로 먼저 봅니다.

```bash
docker compose ps
```

이 명령을 보면:

- 서비스가 실행 중인지
- 어떤 포트로 열렸는지
- 컨테이너 이름이 무엇인지

를 한 번에 확인할 수 있습니다.

## 5. 입문자 추천 학습 순서

아래 순서대로 익히면 포트 관리가 훨씬 덜 헷갈립니다.

### 1단계. 어떤 포트를 쓸지 먼저 확인하기

```bash
cat .env
```

이 프로젝트에서는 기본적으로 `HOST_PORT=8080`입니다.

처음부터 포트 번호를 알고 있어야
뒤에서 `curl`, `lsof`, `docker ps`를 같은 번호 기준으로 확인할 수 있습니다.

### 2단계. 서비스 실행하기

```bash
docker compose up -d
```

초보자는 먼저 서비스가 떠 있는 상태를 만들어야 합니다.
서비스가 실행되지 않은 상태에서 포트만 확인하면 오히려 더 헷갈릴 수 있습니다.

### 3단계. 접속이 되는지 먼저 테스트하기

```bash
curl http://localhost:8080
curl http://localhost:8080/health
```

이 단계에서 배우는 것:

- 서버가 실제로 응답하는지
- 주소를 잘 입력했는지
- 애플리케이션 레벨에서 문제가 없는지

### 4단계. 안 되면 포트가 열려 있는지 보기

```bash
nc -vz localhost 8080
lsof -nP -iTCP:8080 -sTCP:LISTEN
```

이 단계에서 배우는 것:

- 포트 연결 자체가 되는지
- 누가 그 포트를 잡고 있는지

즉, "웹 서버 문제"인지 "포트 문제"인지 구분할 수 있게 됩니다.

### 5단계. 누가 쓰는지 정확히 확인하기

```bash
docker ps
ps -p PID -o pid,ppid,command=
```

이 단계에서 배우는 것:

- Docker 컨테이너가 잡고 있는지
- 로컬 프로그램이 잡고 있는지
- 어떤 프로세스를 종료하거나 설정을 바꿔야 하는지

### 6단계. 충돌 해결하기

선택지는 보통 2가지입니다.

- 충돌 중인 프로세스를 종료한다
- 다른 포트를 사용한다

예:

```bash
kill 1234
```

또는 [`.env`](./.env)의 값을 바꿔:

```bash
HOST_PORT=8081
```

다시 실행합니다.

```bash
docker compose down
docker compose up -d
curl http://localhost:8081/health
```

### 7단계. 마지막으로 실제 연결 구조를 이해하기

이 프로젝트 기준 연결 구조는 아래처럼 이해하면 됩니다.

1. 브라우저 또는 `curl`이 `localhost:8080`에 접속
2. 호스트 `8080` 포트가 컨테이너 `80` 포트로 전달
3. NGINX는 [`compose/nginx/default.conf`](./compose/nginx/default.conf)의 `listen 80;` 설정으로 요청을 받음

이 흐름이 이해되면
"왜 브라우저는 8080인데 NGINX 설정은 80인지"가 자연스럽게 연결됩니다.

## 6. 실전에서 바로 쓰는 문제 해결 순서

초보자는 문제가 생기면 아래 순서대로만 따라가도 훨씬 덜 헤맵니다.

### 상황 1. 브라우저에서 페이지가 안 열린다

1. 어떤 포트를 쓰는지 확인

```bash
cat .env
```

2. 접속 테스트

```bash
curl http://localhost:8080
curl http://localhost:8080/health
```

3. 포트 연결 확인

```bash
nc -vz localhost 8080
```

4. 누가 포트를 쓰는지 확인

```bash
lsof -i :8080
docker ps
```

### 상황 2. "포트가 이미 사용 중" 오류가 난다

1. 어떤 프로세스가 쓰는지 확인

```bash
lsof -nP -iTCP:8080 -sTCP:LISTEN
```

2. PID 상세 확인

```bash
ps -p PID -o pid,ppid,command=
```

3. 종료할지, 포트를 바꿀지 결정

```bash
kill PID
```

또는:

```bash
HOST_PORT=8081
```

4. 다시 실행

```bash
docker compose down
docker compose up -d
```

### 상황 3. Docker는 켰는데 어디로 접속해야 할지 모르겠다

아래 3개를 같이 보면 가장 빨리 감이 잡힙니다.

```bash
cat .env
docker compose ps
docker port compose-web
```

이 3개를 같이 보면:

- 설정 파일에는 어떤 포트가 적혀 있는지
- 실제 컨테이너가 떠 있는지
- 실제로 어떤 포트로 매핑되었는지

를 함께 확인할 수 있습니다.

---

## 7. 초보자가 특히 조심하면 좋은 점

- 컨테이너 안 `80` 포트와 내 컴퓨터의 `8080` 포트를 같은 것으로 생각하지 않기
- `curl`이 안 되는데 브라우저만 계속 새로고침하지 않기
- `kill -9`를 처음부터 쓰지 않기
- 포트를 바꿨다면 접속 주소도 함께 바꾸기
- 문제를 확인할 때는 `설정 확인 -> 접속 테스트 -> 포트 점검 -> 프로세스 확인` 순서를 지키기

## 8. 한 번에 따라 하는 가장 쉬운 실습

이 저장소에서 포트 흐름을 가장 빠르게 익히는 실습은 아래 순서입니다.

```bash
cat .env
docker compose up -d
docker compose ps
curl http://localhost:8080
curl http://localhost:8080/health
lsof -i :8080
docker port compose-web
docker compose down
```

이 실습 하나로 아래 내용을 한 번에 익힐 수 있습니다.

- 프로젝트 기본 포트 확인
- 서비스 실행
- 브라우저 대신 `curl`로 접속 테스트
- 포트를 누가 쓰는지 확인
- Docker 포트 매핑 확인
- 실습 종료 후 정리

핵심은 "포트는 외우는 것"보다
"문제가 생겼을 때 어떤 순서로 확인하는지"를 익히는 것입니다.
